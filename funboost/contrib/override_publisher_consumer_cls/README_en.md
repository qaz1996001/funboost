# 2.1 funboost_otel_mixin.py — OpenTelemetry Full-Chain Task Tracing & Observability, a Production-Grade Strategic Feature of Funboost

To use it, you can either use `OtelBoosterParams` directly,
or specify `consumer_override_cls` and `publisher_override_cls` as `OtelConsumerMixin` and `OtelPublisherMixin` in your `BoosterParams`.

```python
@boost(OtelBoosterParams(
    queue_name='otel_tree_task_entry',
))
```


Funboost's OTel implementation is **exceptionally well-crafted and critically important**. It is the "crown jewel" of Funboost as a modern distributed framework. The author achieves standardized full-chain tracing in a **zero-intrusion** manner through the elegantly designed **Mixin pattern** and **context injection** technique.

## 2.1.1 **Elegant Non-Intrusive Design (Mixin Pattern)**
This also reflects Funboost's architectural flexibility. Rather than modifying the core source code of `Booster` or `AbstractConsumer` to hard-wire tracing logic, the author leverages the `consumer_override_cls` and `publisher_override_cls` interfaces.
*   **Code location**: `funboost/contrib/override_publisher_consumer_cls/funboost_otel_mixin.py`
*   **Usage demo**: `test_frame/test_otel`
*   **Implementation**:
    *   `AutoOtelPublisherMixin`: Overrides `publish` to create a `PRODUCER` span before sending a message, and **injects** the trace context into the message body at `msg['extra']`.
    *   `AutoOtelConsumerMixin`: Overrides `_run` (synchronous) and `_async_run` (asynchronous) to **extract** the context from the message body and create a `CONSUMER` span with it as the parent node.
*   **Advantages**: This design achieves full decoupling. Users who do not need OTel will find no trace of it in the core code (keeping the package size small); those who do need it simply swap in the classes.

### 2.1.2 **Standardized Context Propagation**
This is the hardest and most critical part of distributed tracing.
*   **Producer side**: Uses `opentelemetry.propagate.inject` to embed the current TraceID/SpanID into the message dictionary.
*   **Consumer side**: Uses `opentelemetry.propagate.extract` to restore the context from the message.
*   **Effect**: This ensures the chain `upstream service -> Redis/RabbitMQ -> Funboost consumer` is unbroken. Without this step, the span generated by the consumer would be an isolated new Trace, rendering the tracing meaningless.

### 2.1.3 **Ultimate Extensibility: Architecturally Superior to Celery**
This implementation perfectly demonstrates Funboost's **high extensibility and customizability** at the architectural level.
In Celery, if you want to manually intercept the core pipeline to implement context injection similar to `funboost_otel_mixin.py`, you typically need to study the complex Signal mechanism in depth, define custom Task classes, or even patch the underlying Kombu library — the implementation barrier is extremely high and difficult to maintain.
In Funboost, thanks to the open `override_cls` interface, developers only need standard **OOP inheritance and the Mixin pattern** to cleanly hook into the framework's core flow, enabling any custom requirement from simple logging to complex full-chain tracing.

# 3.1 funboost_micro_batch_mixin.py — Micro-Batch Consumer Mixin

The micro-batch consumer accumulates N messages and then processes them in bulk. It is suitable for scenarios such as bulk database writes and bulk API calls.

*   **Code location**: `funboost/contrib/override_publisher_consumer_cls/funboost_micro_batch_mixin.py`
*   **Usage demo**: `test_frame/test_micro_batch`

## 3.1.1 Usage

```python
from funboost import boost, BoosterParams
from funboost.contrib.override_publisher_consumer_cls.funboost_micro_batch_mixin import MicroBatchConsumerMixin

@boost(BoosterParams(
    queue_name='batch_insert_queue',
    consumer_override_cls=MicroBatchConsumerMixin,
    user_options={
        'micro_batch_size': 100,       # Process after accumulating 100 messages
        'micro_batch_timeout': 5.0,    # Or process after waiting 5 seconds
    },
    should_check_publish_func_params=False, # Must disable parameter validation
))
def batch_insert_to_db(items: list):
    """
    items is a list containing the function arguments of up to 100 messages
    """
    db.bulk_insert(items)
    print(f"Bulk inserted {len(items)} records")
```

## 3.1.2 Core Principles

1. **Buffer accumulation**: Overrides `_submit_task` to accumulate messages into a buffer
2. **Trigger conditions**: Bulk processing is triggered when `batch_size` messages are accumulated or `timeout` seconds have elapsed
3. **Bulk ack/requeue**: On success, messages are bulk-acknowledged; on failure, they are bulk-requeued
4. **Function signature**: The consumer function's parameter changes from a single object to `list[dict]`

## 3.1.3 Applicable Scenarios

| Scenario | Benefit |
|------|------|
| Bulk database writes | Reduces DB connection overhead; throughput improves 10–100x |
| Bulk external API calls | Reduces HTTP connection overhead |
| Bulk notification sending | Merges pushes, reducing request count |

## 3.1.4 Strategic Significance

- Funboost's micro-batch operation is a **production-grade, high-concurrency optimization powerhouse**. It dramatically reduces the complexity of writing batch-processing logic — you don't need to write your own buffer, your own timer, or your own lock handling. Just configure two parameters to upgrade an ordinary consumer into a "batch consumer".
- When you set `Broker` to **`MEMORY_QUEUE`** (Python's native `queue.Queue`) combined with **`MicroBatchConsumerMixin`**, Funboost instantly becomes a **high-performance, in-process, automatic aggregation buffer (In-Memory Batch Aggregator)**.
